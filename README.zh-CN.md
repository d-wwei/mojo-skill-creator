# Mojo Skill Creator

## 为什么做这个？

我们花了几个月搭建和收集 AI Agent 技能——横跨 Claude Code、Codex、Gemini CLI，超过 100 个。规律很快浮出来：大多数通过结构验证的技能，产出仍然是泛泛的。两个不同的任务，同一副骨架，换了一组词。

我们追问为什么。根源不是 prompt 写得不好，也不是结构有问题。而是技能作者从自己的直觉或抽象的「最佳实践」出发设计，从来没研究过领域专家实际怎么做这件事。一个写作技能如果不编码专业作者的思考方式，产出就是 AI 味的文章。一个设计技能如果跳过 Apple、Linear、Stripe 团队真实的工作流程，产出就是安全但没个性的布局。

然后我们找到了两份从相反方向攻击质量问题的工作。[ljg-skills](https://github.com/lijigang/ljg-skills)——14 个持续产出有性格内容的中文技能——证明了定义「不能产出什么」（可机械检验的约束）比告诉 Agent「应该产出什么」更有效。另外，一个 66 品牌设计系统的改进项目证明了技能有可测量的知识层级（泛泛→有场景感知→有辨识度），每层需要不同的升级策略。

缺的那块拼图一旦看到就很明显：**在设计任何约束或工作流之前，先去研究目标领域最好的从业者实际怎么做。** 他们的工作流成为技能的工作流。他们的标准成为技能的红线。他们的思维方式成为技能的姿态。

Mojo Skill Creator 把这些思考打包成了一个可用的工具。它本身是一个技能，用来创建其他技能——也升级已有的技能——每一步都内置了这些理念。

## 是什么

Mojo Skill Creator 本身是一个 AI Agent 技能（SKILL.md + 8 个 reference 文件，总计约 8,500 词）。安装到任何支持 SKILL.md 格式的 Agent 上，它就获得两个能力：

- **`new`** — 9 步引导流程，从零创建技能，从领域专家研究开始
- **`boost`** — 4 阶段诊断升级流程，针对已有技能

产出的每个技能都跨平台（Claude Code、Codex、Gemini CLI、OpenClaw）且 Token 高效。

## 关键特性

- **领域研究先行。** 在写任何红线或工作流步骤之前，引导你去研究最好的从业者实际怎么做——他们真实的工作流、质量标准和原则。然后要求你重新思考：技能需要不同的工作流结构吗？原有的假设还成立吗？质量标准到底应该是什么？这一步是"从通用零件拼装的技能"和"用真实专业知识灌注的技能"之间的分水岭。

- **约束驱动的质量。** 每个技能出厂自带 5-10 条可机械检验的红线——产出不能包含的东西。「每句话不得包含超过一个从句」可以逐句扫描验证。「写得清楚一点」不行。来自专家标准的红线能捕获抽象指导遗漏的失败模式。

- **知识层级诊断。** `boost` 命令诊断技能所在的层级：原则层（正确但泛泛）、模式层（有场景感知但没辨识度）、案例层（有独特性，能发现自己的错误）。每层有不同的升级路径。不浪费力气把案例层的技术用在原则层的技能上。

- **四层 Token 架构。** 技能不把所有东西塞进一个文件。元数据(~100词) → 路由(≤1000词) → 工作流(≤2000词) → 参考资料(按需)。每次调用只加载一个工作流。

- **默认跨平台。** 指令使用语义化动词（「读取文件」「搜索代码库」），不用平台特有工具名。一个技能包在四个平台上直接可用，不需要适配。

- **可选自进化。** 技能可以打包 [skill-se-kit](https://github.com/d-wwei/skill-se-kit)（~48KB）来从自身使用中学习。每次任务完成后，子 Agent 提取反馈、记录经验、更新技能库——技能随使用自动改进，无需手动干预。se-kit 打包在技能内部，终端用户不需要额外安装。技能作者通过 `boost` 更新打包版本后重新分发。

## 用与不用的对比

| | 不用 Mojo | 用 Mojo |
|---|---|---|
| **起点** | 作者直觉 + 通用模板 | 领域专家的工作流 + 质量标准 |
| **红线** | 模糊（「写清楚」）或没有 | 可机械检验，来源于专家实践 |
| **工作流设计** | 从零发明 | 镜像最好的从业者实际怎么做 |
| **质量诊断** | 「感觉很泛」 | 可测量：原则层 / 模式层 / 案例层，各有升级路径 |
| **Token 开销** | 每次全量加载 | 只加载当前调用需要的工作流 |
| **平台支持** | 跑在作者的平台上 | Claude Code、Codex、Gemini CLI、OpenClaw |
| **运行时学习** | 静态——部署后不会改进 | 可选自进化：技能从使用中学习，积累领域经验 |

## 怎么工作

### `new` — 创建技能（9 步）

1. **理解使用场景** — 收集 3-5 个具体示例，包括失败场景
2. **研究领域最佳实践** — 找到最好的从业者，研究他们实际的工作流和质量标准，做跨学科扫描，然后综合：重新思考工作流、原理和质量标准。详细指南在 `references/domain-research-guide.md`
3. **设计红线和验收标准** — 从专家标准导出 5-10 条约束、3-5 条可测试标准、定义姿态（认知位置而非角色身份）——全部在写内容之前
4. **规划 Token 架构** — 设计四层结构，每层分配词数预算
5. **规划可复用资源** — 确定 scripts/、references/、assets/ 的内容
6. **构建技能** — 写 SKILL.md（路由）、工作流 reference、深度 reference。工作流结构镜像第 2 步发现的专家流程
7. **人类可审计**（可选）— 添加 artifact 输出让执行过程可追溯
7b. **自进化**（可选）— 打包 skill-se-kit（~48KB）实现运行时学习
8. **跨平台验证** — 无平台工具名、所有引用存在、预算达标
9. **迭代** — 观察首次使用，收紧红线、澄清工作流

### `boost` — 升级已有技能（4 阶段）

**Phase 1：诊断** — 8 项检查：结构审计、知识层级诊断、上下文效率、人类可审计性、跨平台兼容、自进化状态、领域最佳实践研究（对比技能工作流 vs 专家工作流）、研究综合（修补还是重构？）。

**Phase 2：处方** — 架构修复 → 机械修复 → 内容升级，严格按序。如果研究综合发现工作流存在结构性差距，处方从工作流重构开始，不是内容润色。

**Phase 3：执行** — 按序修复，每类修完后验证。

**Phase 4：报告** — 人类可审计的升级总结，含前后对比指标。

### 设计决策

| 选择 | 为什么 |
|------|--------|
| 先做领域研究再做设计 | 基于专家工作流设计的技能产出更贴合领域。实践验证：研究 Apple/Linear/Stripe/Airbnb 设计实践后，特异性测试通过率从 ~20% 升到 ~85%。 |
| 约束先于流程 | 受 ljg-skills 影响。可检验的约束（「每句不超过一个从句」）比理想化指导（「写好一点」）能捕获更多失败模式。 |
| 姿态替代角色 | 「不教课，不聊天——先亮弯路，再给方向」比「你是资深写作教练」产出的行为更精确。 |
| 四层 Token 架构 | 大多数技能每次调用都加载大量无关材料。分层让每次调用保持精简。 |

## 安装

```bash
git clone https://github.com/d-wwei/mojo-skill-creator.git
ln -sf "$(pwd)/mojo-skill-creator" ~/.claude/skills/mojo-skill-creator
```

对 Agent 说：「创建一个新技能」或「升级这个技能」。

### 支持平台

| 平台 | Skill 路径 |
|------|-----------|
| Claude Code | `~/.claude/skills/` |
| Codex | `~/.agents/skills/` |
| Gemini CLI | `~/.gemini/skills/` |
| OpenClaw | `~/.openclaw/skills/` |
