# ADR-0001: Adopt Development Practices and ADR Mechanism

## Status
Accepted (2026-04-17)

## Context
This is a retroactive record. The decision to adopt Development Practices and ADRs was made during a structured brainstorm/plan cycle (docs/brainstorms/development-principles-requirements.md).

MSC had 7 design philosophy principles and 10 red lines, but no codified engineering practices — rules for how development is conducted (dependency management, backward compatibility, decision recording, testing discipline, etc.). The user proposed 13 concrete principles drawn from production experience. These needed a structured home in the skill.

Separately, architecture decisions and "decided not to do" decisions had no durable record format. Rationale lived in brainstorm artifacts or conversation history, which decay across sessions.

## Decision
1. Add a "Development Practices" section to SKILL.md with 10 universal principles (thin declarations, ≤20w each)
2. Create `references/development-practices.md` with full principle details, including conditional domain-specific practices (P9 security layering, P11 anti-duplicate counting)
3. Adopt ADR (Architecture Decision Records) as the mechanism for recording architecture decisions and non-decisions
4. Promote 3 practices to Red Lines: ADR requirement (P4+P5), backward compatibility (P6), secrets safety (P10)
5. Upgrade CHANGELOG to hybrid format: narrative header (≤5 lines) + Keep a Changelog categories
6. Distribute enforcement via single-line checkpoints in `new-workflow.md` and `boost-workflow.md`

ADR threshold: decisions affecting ≥2 files or constraining future changes.

## Rejected Alternatives
- **Expand Core Design Principles to include all 13**: Rejected because design philosophy (how to design skills) and engineering practices (how to develop software) are different categories. Mixing them dilutes the clarity of both.
- **Pure reference file with no SKILL.md presence**: Rejected because on-demand-only loading means principles would be invisible during most workflow execution.
- **Build a principles engine with automated checking**: Rejected because the scale (11 principles) doesn't justify dedicated infrastructure. Existing stage gates and review personas provide sufficient enforcement.

## Consequences
- SKILL.md grows by ~225w (from 528w to ~753w), still under 1000w router budget
- `new-workflow.md` requires ~100w compression to fit 6 checkpoints (file is at 1994/2000w)
- ADR directory introduces a new artifact type — future decisions should be recorded here
- Verification scripts for P7/P8 Do-axis enforcement are deferred to a follow-up iteration
- MSC itself does not yet have CI (P7 Level A is aspirational until CI is set up)
