# Iteration Roadmap v3

## 当前状态速览

v3 已交付。新增统一约束框架（Think + Do 双轴模型）。SKILL.md 10 红线（+1），5 验收标准（+1），7 设计原则（+1），9 reference 文件（+1），3 验证脚本（新增）。Token 预算全部达标。

## 本次迭代经验

### 做对了
- Dogfood 路线（先建方法论，再对自身应用）产出了高质量的自我分析和具体改进
- 压缩先行策略有效：先压缩再新增，每步都在预算内，避免了"先加再压"的风险
- 对抗性审查发现了关键元矛盾（消除 Think-only 的框架本身是 Think-only），避免了盲目自信

### 需改进
- 所有 artifact gate 仍然是认知约束——结构性强制受限于跨平台兼容。Hook 模板是下一步，但需要平台分层策略
- 验证脚本是自愿执行的——无法证明脚本真正跑过。未来需要 evidence file 机制
- 30% Do-axis 目标在 review 中被指出可被 stakes 分类游戏化——已通过脚本加强但根因（自评 stakes）未解

### 新发现
- **Think + Do 双轴是有效的思维模型**，但 Do 轴的天花板取决于平台能力。当前四平台只有 Claude Code 完整支持 hooks。
- **元矛盾**：消除 Think-only 约束的框架本身用 Think-only 约束执行。这不是 bug 而是指令系统的根本限制——值得深入研究突破口。
- **压缩的最佳时机是新增之前**，不是之后。这次先量化现有文件超标情况，再有针对性压缩，效果比事后补救好得多。

## 建议的下一个迭代

### ✅ Priority 1: 统一约束框架集成
- 已完成：方法论 + 自我分析 + 实施 + 验证脚本

### Priority 2: 实战验证 — 用 `new` 创建一个真实 skill（含 Step 3d）
- **描述**: 用 mojo-skill-creator 的完整 `new` 流程（包括新增的 Step 3d）从零创建一个 skill，验证约束执行设计步骤是否有效
- **验证**: 创建的 skill 有 `build/constraint-enforcement-plan.md`，≥30% 红线有 Do 机制

### Priority 3: Hook 模板参考实现
- **描述**: 为 MSC 自身和产出 skill 创建 `hooks/` 目录，提供 Claude Code PreToolUse hook 示例。作为可选增强，不影响跨平台兼容
- **验证**: hooks/ 目录存在，含至少 1 个 PreToolUse hook 模板，文档说明如何安装

### Priority 4: Evidence file 机制
- **描述**: 验证脚本写入 `build/.verify-*.result` 时间戳文件，后续 gate 检查 result 文件的存在和新鲜度
- **验证**: 脚本产生 evidence file，workflow 中的 gate 检查这些 file

### Priority 5: 实战验证 — 用 `boost` 升级一个已有 skill（含 Phase 1.9）
- **描述**: 选一个已有 skill，用完整 `boost` 流程（包括新增的 Phase 1.9）诊断和升级
- **验证**: `diagnosis/constraint-enforcement-audit.md` 存在，审计比例有意义
