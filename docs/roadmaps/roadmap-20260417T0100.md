# Iteration Roadmap v4

## 当前状态速览

v4 已交付。新增 Development Practices 工程纪律层（10 通用原则 + 2 条件原则）、ADR 机制、混合 CHANGELOG 格式。SKILL.md 13 红线（+3），5 验收标准，7 设计原则，11 reference 文件（+1），3 验证脚本。Token 预算全部达标，new-workflow.md 在 2000/2000w 天花板。

## 本次迭代经验

### 做对了
- Thin-declaration + distributed-enforcement 模式：SKILL.md 只加 ~150w，enforcement 分布在 workflow 步骤中，不膨胀 router 层
- 压缩先行（延续 v3 策略）：new-workflow.md 从 1994w 压缩到 1894w 后再加 checkpoint，最终 2000w
- 对抗性审查两轮（brainstorm + review）：分别发现 token 预算矛盾和 P-ID 一致性问题，全部在 ship 前修复
- ADR 作为 bootstrapping 记录：ADR-0001 记录引入 ADR 机制本身的决策，消除递归困惑

### 需改进
- new-workflow.md 在 2000/2000w 天花板，未来任何添加都需要先压缩或拆分文件
- boost-workflow 没有 "Dev Practice" 文本格式 checkpoint（用的是诊断表格行），与 new-workflow 格式不一致
- P2（测试）和 P13（用户行为调查）的 enforcement 是隐含的（Step 10 和 Step 1），没有显式 "Dev Practice P2/P13" 标记
- 验证脚本未更新（P7 linter、P8 CHANGELOG 格式的 Do-axis enforcement deferred）

### 新发现
- **表格格式压缩 ~60%**：reference 文件用表格替代散文后从预估 1450-2150w 降到 879w，省出大量空间
- **原则数量的数学陷阱**：requirements 说 "9 条" 但枚举出 10 行，P4+P5 合并导致计数歧义。以后在 requirements 中统一用枚举而非抽象计数
- **Level A/B 分类价值**：区分 "MSC 自身" 和 "产出 skill" 避免了在通用 skill 中硬编码 Rust-specific 规则
